## 연구 배경 및 문제

### 대규모 클라우드 네트워크의 장애 관리 현실
- Alibaba Cloud는 전 세계 29개 리전, 89개 데이터센터, $O(10^5)$개 네트워크 장비를 운영
- 장애 원인이 다양함: 디바이스 하드웨어 오류(42.6%), 링크 오류(18.5%), 네트워크 변경 오류(16.7%), 인프라 오류(9.3%), 디바이스 소프트웨어 오류(9.3%), 라우팅 오류(1.9%), 보안 오류(1.9%), 설정 오류(1.9%)
- 분(minute) 단위 장애 복구 SLA를 목표로 하지만, 심각한 장애 시 수 시간 소요되는 경우 발생

### 기존 모니터링 도구의 한계: 단일 데이터 소스 의존
각 모니터링 도구는 고유한 커버리지 한계를 가짐 (개별 도구 커버리지: 3%~84%):

| 데이터 소스 | 한계 |
|-----------|------|
| **Ping** | 도달성(reachability)만 감지, 이중화에 영향을 주는 부분적 링크 장애는 탐지 불가 |
| **Traceroute** | 비대칭 경로 또는 SRTE 터널 사용 시 효과 상실 |
| **Syslog** | 디바이스가 인지하지 못하는 장애(silent packet loss, 전송 오류, 라우팅 오류, bit flip) 감지 불가 |
| **SNMP** | SNMP 프로토콜 내 수집 가능 정보로 제한 |
| **Out-of-band** | 인프라 관련 이슈(디바이스 활성, CPU, 온도 등)에만 집중 |
| **INT (In-band Network Telemetry)** | 모든 디바이스에서 지원되지 않음 |
| **Route monitoring** | 컨트롤 플레인에 국한, 데이터 플레인 이슈 진단 불가 |

### "Alert Flooding" 문제
다수의 모니터링 도구를 통합하면 커버리지는 높아지지만, **심각한 장애(severe failure)** 시 발생하는 핵심 문제:

1. **대량 알림 폭주**: 심각한 장애는 다수의 장비와 링크에 영향 → 수 분 내 $O(10^4)$개 이상의 알림 생성 → 운영자가 분 단위 SLA 충족 불가
2. **전례 없는 장애("unknown failures")**: 심각한 장애는 드물고 점점 더 많은 수가 전례 없는 유형 → 사전 정의된 SOP(Standard Operating Procedure)나 모델에만 의존 불가
3. **동시 발생 경미 장애**: 예정된 업데이트나 경미한 장애의 알림이 동시 발생 → 수동 근본 원인 분석 방해

### 실제 사례
- 데이터센터 인터넷 입구 케이블의 절반이 동시 장애 → 10,000개 이상 알림 발생
- 운영자가 패킷 손실을 디바이스 장애로 오인 → 디바이스 격리했으나 문제 지속
- 실제 원인은 혼잡(congestion)이었으나, SNMP의 혼잡 알림이 알림 폭주에 묻힘
- 완화까지 수 시간 소요, 막대한 재정·평판 손실

### LLM 기반 진단이 부적합한 이유 (§2.3)
- Syslog만으로도 15분당 약 1,000만 개 로그 생성 → 최대 2,000만 토큰을 지원하는 long-context LLM의 입력 용량 초과
- LLM은 블랙박스로 동작하며 hallucination 위험 → 오진 시 운영자가 진단을 처음부터 재시작해야 함
- 현재 LLM은 보조 도구로만 활용 가능, 심각한 장애의 독립적 처리에는 부적합

### 운영자가 필요로 하는 정보
- **장애 발생 사실**, **관련 알림 (특히 알림의 유형)**, **장애의 심각도**
- 원래 $O(10^4)$개 알림 대신 약 10개의 정제된 메시지를 제공하면 수동 제어 가능

## 설계 상세

### 4.1 Preprocessor (전처리기)

모니터링 도구 간 차이를 표준화된 구조(Timestamp, Location, Type)로 변환:

#### 세 가지 차이점 해소

**1. 알림 주기 차이**
- Ping: 2초마다 데이터 포인트 생성 vs Syslog: 오류 발생 시에만 로그
- 해결: 유사 알림 집계. 예) Ping 패킷 손실 시작 시간을 알림 타임스탬프로, 후속 알림은 "duration" 속성에 기여

**2. 위치 정보 구조 차이**
- Syslog/SNMP: 소스 디바이스가 명확 vs Ping: 두 논리 사이트 간 패킷 손실 → 중간 링크에 귀속
- 해결: 네트워크 계층 구조(Region → City → Logic Site → Site → Cluster) 활용
  - 각 디바이스는 계층의 한 수준에 배정
  - 링크 관련 알림은 연결하는 두 디바이스에 대한 알림으로 분리
  - 상위 디바이스 장애 시 하위 모든 디바이스 영향받음 → 장애 범위 평가에 활용

**3. 알림 내용 차이**
- Ping, SNMP: 제한적 알림 유형 → 수동 정의
- Syslog: 수천 종의 CLI 출력 → **FT-tree** 기반 템플릿 자동 분류
  - CLI 출력을 단어로 분해 → 정규표현식으로 변수 단어 제거 → 나머지로 템플릿 생성
  - 중요도 순으로 수동 타입 할당 (수개월에 걸쳐 완료)

#### 알림 수 축소를 위한 3가지 통합 방법

| 방법 | 설명 |
|------|------|
| **동일 알림 통합** | 같은 도구에서 중복 알림 수신 시, 이미 분석된 알림은 무시하고 타임스탬프만 갱신 |
| **단일 소스 내 통합** | 임계값 미달 알림 무시 (예: 산발적 패킷 손실 무시, 지속적 손실만 기록), 연관 알림 필터링 (예: 한 인터페이스 트래픽 급증 → 인접 인터페이스 트래픽 증가 알림 제거) |
| **다중 소스 간 통합** | 단독으로는 문제를 나타내지 않지만, 다른 도구 데이터와 결합 시 이슈를 시사하는 알림 식별 (예: 트래픽 급감 단독은 정상이나, 패킷 손실 또는 Syslog 오류와 함께 발생 시 비정상) |

### 4.2 Locator (위치 탐지기)

#### 핵심 인사이트
- 같은 위치·같은 시간대의 다중 알림 → 네트워크 인시던트 가능성
- 네트워크 알림은 토폴로지 링크를 따라 전파 → 인접 장비 동시 알림은 같은 근본 원인
- 운영자는 장애의 전체 범위를 파악해야 범위·심각도 평가 가능

#### 계층적 Alert Tree ("Main Tree")

네트워크 계층 구조(Region → City → Logic Site → Site → Cluster)에 따라 구성된 트리에 알림을 시간·위치 기준으로 삽입:

**Algorithm 1 - Alert Node Insertion:**
- 새 알림 도착 시, 기존 인시던트 트리 또는 메인 트리에서 해당 위치 노드 검색
- 위치 노드가 존재하면 알림 추가, 없으면 새 노드 생성

**Algorithm 2 - Incident Tree Generation:**
- 각 노드의 하위 트리 알림 수 카운트 (연결된 영역 내 알림만 계산)
- 인시던트 생성 임계값: **failure 알림 > 2** 또는 **failure 알림 ≥ 1 AND 전체 알림 > 3** 또는 **전체 알림 > 5**
- 임계값 초과 시 해당 하위 트리를 메인 트리에서 복제하여 "인시던트 트리"로 지정
- 기존 하위 인시던트 트리가 있으면 상위 인시던트로 병합

**Algorithm 3 - Tree Checking (타임아웃):**
- 메인 트리 노드: 5분 초과 시 제거 (SNMP 최대 지연 ~2분 + 전송 오류 대비)
- 인시던트 트리: 15분간 새 알림 없으면 종료

#### 알림 카운팅 방식
- **주파수 차이 해소**: 같은 타입의 다수 알림은 1개로 카운트 (타입 기반 카운팅)
- **중요도 차이 해소**: 알림을 3단계로 분류

| 수준 | 설명 | 예시 |
|------|------|------|
| **Failure 알림** | 네트워크 동작이 확정적으로 비정상 | 패킷 손실, 패킷 bit flip, 높은 전송 지연 |
| **Abnormal 알림** | 불규칙한 동작이지만 직접적 장애 의미는 아님 | 지터, 지연 급증, 트래픽 급감 |
| **Root cause 알림** | 네트워크 엔티티의 장애 표시 | 디바이스/NIC 장애, 링크 다운, CRC 오류, 위험 라우팅 경로 |

- **Failure 알림이 인시던트 탐지에 가장 중요**: 전체 알림 중 비율은 낮지만, 거의 모든 장애 인시던트에 동반됨 (Figure 5d)
- **False alarm 방지**: 같은 타입의 알림은 출처 장비와 무관하게 1개로 통합 (예: 장치 프로브 오류 → 대량의 동일 device down 알림 → 1개로 집계)

#### 인시던트 보고서 출력 예시
```
Incident 1: [Region A | City a | Logic site 2][시작-종료 시간]
  Failure alerts:
    Ping
    ├─ End to end ICMP (3)
    ├─ End to end Source (1)
    └─ End to end TCP (1)
  Abnormal alerts:
    Out-of-band
    └─ Inaccessible (680)
    Syslog
    ├─ Traffic blackhole (1)
    ├─ Link flapping (8)
    ├─ Port flapping (3)
    └─ BGP peer down (2)
  Root cause alerts:
    Syslog
    ├─ BGP link jitter (4)
    ├─ hardware error (1)
    └─ out of memory (2)
    SNMP
    ├─ Traffic congestion (1)
    └─ Link down (6)
```

### 4.3 Evaluator (평가기)

동시 다발 인시던트의 우선순위를 정량적으로 결정.

#### 심각도 평가 기준

| 기준 | 설명 |
|------|------|
| **Packet Loss Ratio** | Ping 및 sFlow 기반 패킷 손실 비율. 트래픽 조건에 따른 정규화를 위해 손실률(rate)이 아닌 손실 비율(ratio) 사용 |
| **Link Break Ratio** | 이중화 링크 세트 중 단절 비율. 직접적 패킷 손실은 아니지만 총 대역폭 감소 → 트래픽 급증 시 서비스 품질 저하 |
| **Importance of Affected Customers** | 높은 안정성 SLA를 구매한 고객 우선. Netflow 기반 영향 산정 |
| **Duration of Incident** | SLA에 따라 복구 시간이 길어질수록 재정·평판 손실 증가 |

#### 심각도 계산 공식

**Impact Factor (영향 계수):**

$$I_k = \max\left(1, \sum_{i=1}^{N} d_i g_i u_i + \sum_{j=1}^{N} l_j g_j u_j\right)$$

- $N$: 인시던트 관련 회선 세트(circuit set) 수
- $d_i$: 회선 세트 $i$의 단절 비율, $l_i$: 회선 세트 $i$에서 SLA 초과 플로우 비율
- $g_i$: 고객 중요도 계수, $u_i$: 고객 수
- max(1, ...) → 중요 고객이 없더라도 심각도가 0이 되지 않도록 보장

**Time Factor (시간 계수):**

$$T_k = \max\left(\log_{\frac{1}{R_k}}(\Delta T_k + Sig(U_k)),\ \log_{\frac{1}{L_k}}(\Delta T_k + Sig(U_k))\right)$$

- $R_k$: 평균 Ping 패킷 손실률, $L_k$: 최대 평균 SLA 플로우 초과율
- $\Delta T_k$: 알림 지속 시간, $U_k$: 중요 고객 수
- 시그모이드 함수($Sig$): 소수 핵심 고객 영향 시 심각도 크게 반영, 다수 영향 시 안정화 → 지터로 인한 오경보 방지
- 높은 패킷 손실률 → 시간에 따른 심각도 성장 가속

**최종 심각도:**

$$y_k = I_k \cdot T_k$$

#### Location Zoom-in (위치 정밀화)

인시던트 영역 내 정확한 장애 위치를 식별하기 위한 세 가지 조건:

| 트리거 | 방법 |
|--------|------|
| **Ping 도달성 행렬** | End-to-end Ping 결과로 도달성 행렬 구축 → 높은 패킷 손실의 행·열 교차점이 장애 위치 (클러스터~리전 단위 가변 세분도) |
| **sFlow 기반** | sFlow 패킷 손실 감지 시, 영향받은 모든 장비가 인시던트 트리의 특정 노드로 귀결 → 해당 노드로 위치 정밀화 |
| **INT 기반** | 특정 DSCP 값의 테스트 플로우로 입출력 비율 비교 → 차이 발생 시 패킷 손실 의미, 인시던트 트리 노드와 매칭하여 위치 정밀화 |

## 배포 경험 및 사례 연구

### 사례 1: Known Failure 자동 SOP
- 한 네트워크 장비에서 패킷 손실, Syslog 오류, 비정상 SNMP 카운터 알림 감지
- 같은 그룹 내 다른 장비는 알림 없음
- SkyNet이 해당 장비만의 인시던트 보고서 생성 → 자동 SOP 발동 → 장비 격리
- **약 1분 만에 완화 완료**, 수동 개입 없음

### 사례 2: 다중 장면 탐지 (DDoS)
- 5곳 동시 DDoS 공격 발생
- SkyNet이 위치별로 5개의 별개 인시던트 생성 → 공격이 서로 무관함을 즉시 파악
- 5개 리전 모두에 대해 신속하고 포괄적으로 IP 차단 실행
- SkyNet 없이는 일부 공격 지점을 놓칠 수 있었음

### 사례 3: 장면 순위 결정
- 2개의 네트워크 장애가 거의 동시 발생
- 하나의 인시던트가 더 넓은 지역·더 많은 알림을 포함했으나, SkyNet은 **트래픽·서비스 중요도 기반으로 다른 인시던트를 더 긴급하다고 판단**
- 운영자가 더 중요한 인시던트 우선 처리 가능

### 사례 4: Fine-grained Localization
- §2.2의 사례와 유사한 인터넷 입구 케이블 장애 재발
- SkyNet이 관련 알림을 단일 인시던트로 통합 + Ping 도달성 행렬로 데이터센터 입구를 장애 지점으로 특정
- 케이블 수리 포함 **수 분 만에 완화** (이전 사례 수 시간 대비 약 2자릿수 단축)

## 성능 평가

### 실험 환경
- 서버: Intel Xeon Platinum 8260 CPU (2.40GHz), 96코어, 32GB RAM
- 데이터셋: 1.5년간 실제 프로덕션 네트워크($O(10^5)$ 장비)에서 수집된 알림, 10종 이상의 모니터링 도구

### 6.1 커버리지
- 데이터 소스 수를 체계적으로 줄여 평가 (커버리지 낮은 것부터 제거)
- 데이터 소스 감소 시: false positive 영향 미미, **false negative 증가** (미감지 장애 발생)
- 결론: **가능한 많은 데이터 소스 통합이 포괄적 커버리지에 필수적**

### 6.2 전처리 효과
- **전처리 전**: 시간당 약 100,000개 알림
- **전처리 후**: 정상 시 < 10,000개, 극단적 상황에서도 < 50,000개
- **Locator 처리 시간**: 최악의 경우에도 <10초 (분 단위 SLA 대비 충분)
- 전처리 없이는 Locator가 수 분 소요 → 비효율적

### 6.3 정확도
- 프로덕션 파라미터: **"2/1+2/5"** (failure 2개 / failure 1개 + 기타 2개 / 전체 5개)
- 현 설정에서 **false negative 0%** 달성, false positive 최소화
- 타입+위치 구분 없이 처리 시: false positive 20% → 70% 증가
- 임계값 중 하나라도 비활성화 시: false negative 증가

### 6.4 Evaluator 효과
- 장애 관련 인시던트의 심각도 점수가 전체 인시던트보다 일관되게 높음
- 심각도 임계값 10 설정 → 인시던트 수 약 **100배 감소** (false negative 없이)
- 필터 후 평균 인시던트 수: **하루 1건 미만**

### 전체 성과
- 중간값 완화 시간: 736초 → **147초** (80% 감소)
- 최대 완화 시간: 14,028초 → **1,920초** (86% 감소)
- 1.5년간 가동, **false negative 0건**, 80% 이상의 장애에서 완화 시간 단축

## 시각화 (§7.1)
- SkyNet 프론트엔드: 노드=네트워크 장비, 에지=연결
- 인시던트의 시간·공간 범위로 그래프 한정
- **Alert voting** 기반 하이라이팅: 장비/링크가 생성한 알림이 자신 및 연결된 장비/링크에 투표
- 투표 점수 최고 장비 → 근본 원인 가능성 높음 (예: reflector 장비 식별 → 즉시 완화)

## 교훈 및 논의

### 휴리스틱 규칙 vs SkyNet
- 약 1,000개의 휴리스틱 규칙 운용 중이나, 전례 없는 장애 대응 불가
- 현재 운용 구조: **경미한 장애 → 휴리스틱 시스템**, **심각한 장애 → SkyNet**

### 알림 간 시계열 관계의 한계 (§7.3)
- 직관적으로 첫 번째 알림이 근본 원인처럼 보이지만, 실제로는 **동작 이상 알림이 먼저, 근본 원인 알림이 나중에** 수집되는 경우가 빈번
- 사례: BGP 링크 단절 알림 → 패킷 드롭·장비 접근 불가 알림 폭주 → 수 분 후 하드웨어 오류 Syslog 수신 (이것이 실제 근본 원인)
- **결론: 시계열 기반 인과 관계 분석 대신, 타임아웃 윈도우 기반 Alert Tree로 알림 연관 및 인시던트 탐지**
