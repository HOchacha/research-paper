## 1. 연구 배경 및 문제 정의

### 클라우드 네이티브 모니터링의 필요성
- 클라우드 서비스가 모놀리식(monolithic)에서 마이크로서비스 기반 아키텍처로 전환됨
- SLA(Service Level Agreement) 보장을 위해 시스템/애플리케이션 메트릭 모니터링이 필수적

### 기존 모니터링 시스템의 2가지 핵심 문제

| 문제 | 설명 |
|------|------|
| **서비스 간섭** | 기존 모니터(Netdata, Prometheus 등)가 호스트 CPU를 점유하여 서비스 성능 저하 유발. 알리바바 "쌍십일(双十一)" 쇼핑 페스티벌에서 모니터링이 온라인 서비스에 지터/성능 저하를 일으킴 (처리량 6.25% 감소, 꼬리 지연 2배 증가) |
| **모니터링 QoS 미보장** | 높은 시스템 부하나 제한된 CPU 쿼터 하에서 모니터링 지연이 10배 이상 증가. 대규모 메트릭의 빠른 변화 추적 불가 |

### 추가 도전과제: 네트워크 Incast
- RDMA 기반 모니터링에서 컨트롤러가 다수 에이전트로부터 동시에 데이터를 수신 → **심각한 incast 문제**
- PFC/ECN 메커니즘 트리거 → 처리량이 대역폭의 10% 이하로 감소, 꼬리 지연 2~20배 증가
- 클라우드 네이티브 환경의 탄력적 서비스 배포 + 블랙박스 네트워크로 인해 흐름 제어가 더욱 어려움

## 2. ZERO+ 시스템 설계

### 핵심 아이디어
- **One-sided RDMA READ**를 활용하여 모니터링 대상 호스트의 **CPU 점유율 제로**로 메트릭 수집
- **수신자 주도(Receiver-Driven) 모델**: 컨트롤러가 READ 요청을 발행하여 메트릭을 능동적으로 가져옴

### 메트릭 특성 활용
1. 대부분의 모니터링 메트릭은 **카운터**로, 초기화 후 고정 메모리 영역에서 업데이트됨
2. 원시 메트릭 처리는 **단순 대수 연산**으로 컨트롤러에 오프로드 가능

### 아키텍처 개요

```
┌─────────────────────────────────────────────┐
│            Host (PM/VM)                      │
│  ┌──────────┐   ┌──────────────────────┐    │
│  │ ZERO+    │   │ Shared Memory        │    │
│  │ Agent    │──▶│ MetaData  | Metrics  │    │
│  │(Blocking)│   │(Control)  | (Data)   │    │
│  └──────────┘   └──────────────────────┘    │
│                        │                     │
│                   QP Connection              │
└────────────────────────┼─────────────────────┘
                         │ RDMA WRITE / READ
┌────────────────────────┼─────────────────────┐
│              ZERO+ Controller                 │
│  ┌─────────────────┐ ┌─────────────────────┐ │
│  │ Collecting Thread│ │ Processing Thread   │ │
│  │ Flow Control +   │ │ Persistence +       │ │
│  │ Hybrid I/O       │ │ Visualization       │ │
│  └─────────────────┘ └─────────────────────┘ │
└───────────────────────────────────────────────┘
```

## 3. 제어 평면 (Control Plane) & 데이터 평면 (Data Plane)

### 제어 평면 (일회성 오버헤드)
- 에이전트가 시스템/애플리케이션의 메모리 영역(MR)을 RNIC에 등록
- 메타데이터(주소, rkey, 타입, 크기)를 **제어 영역(Control Region)** 에 기록
- 등록 후 에이전트는 **블로킹 모드**로 진입 → CPU 점유 없음
- 서버리스 시나리오에서는 RDMA WRITE를 사용하여 원격 제어 영역에 메타데이터 직접 쓰기

### 데이터 평면 (제로 오버헤드)
- `mmap`으로 메트릭 메모리 영역을 에이전트 프로세스 → 원격 컨트롤러에 노출
- 컨트롤러가 **RDMA READ**로 데이터 영역 직접 읽기 → **CPU 점유 0, 메모리 복사 0**
- 읽기-쓰기 일관성: 원자적 변수(1~3ns)는 RDMA 연산(1μs)보다 3자릿수 빠름 → 일관성 보장
- 비원자적 변수: 비트 플래그로 업데이트 상태 표시

### 메모리 관리 (필수)
- 메트릭이 분산된 메모리에 위치하면 MR 수 증가 → RNIC SRAM 축출 → 성능 저하
- **방법 1**: 변수 속성(`__attribute__((section(".zero_init")))`)으로 같은 데이터 세그먼트에 배치
- **방법 2**: ZERO+ 메모리 할당자로 연속 공간에 할당
- 결과: 하나의 READ로 메트릭 리스트 전체를 가져올 수 있음

### 인터페이스
```c
void *reg_mr(void *addr, const size_t length, int key);  // 등록
void *dereg_mr(int key);                                   // 해제
void *zero_init();                                         // 초기화
struct disk *my_disk = zero_malloc(sizeof(struct disk));   // 할당
```

## 4. 흐름 제어 (Flow Control)

### 수신자 주도 모델의 이점
1. 컨트롤러가 필요에 따라 READ 요청 발행 → 대상 지연/업데이트 빈도 충족
2. 동시 플로우의 총 인플라이트 데이터 제한으로 혼잡 회피 용이

### Credit 기반 흐름 제어 (Alpha 버전, 한계 있음)
- 대형 READ를 4KB 페이지 단위로 분할, 크레딧으로 인플라이트 데이터 제한
- **문제**: 최적 크레딧 값이 모니터링 규모와 네트워크 환경에 따라 달라짐 → 고정 설정 불가

### 대역폭 공유 흐름 제어 (최종 설계)
- 병목 대역폭 $BW$를 에이전트 수 $Num$에 따라 균등 분배:

$$BW_i = \frac{BW}{Num}$$

- READ 요청 발행 시간 비용 $T_{cost}$ ≈ 0.5~1.5μs/에이전트
- 시간 간격: $T = Num \times T_{cost}$
- 각 에이전트의 1회 전송량: $BytesToRead = BW_i \times T$
- **예시**: BW=10GB/s, Num=300 → BWᵢ=0.033GB/s, T≈300μs, BytesToRead=15KB
- 동적 시간 간격 T로 네트워크 환경 변화에 적응

### 우선순위 큐 (QoS)
- 모니터링 요청을 High/Middle/Low 우선순위 링크드 리스트로 구분
- 컨트롤러가 높은 우선순위 에이전트에 먼저 READ 요청 발행
- 1k 호스트 모니터링 시 고우선순위가 저우선순위보다 **400μs 이상** 빠르게 완료

## 5. 컨트롤러 측 최적화

### 스레드 분리
- **수집 스레드 1~2개**: READ 요청 발행 + 완료 폴링 (배치 처리)
- **처리 스레드 다수**: 원시 메트릭을 struct로 파싱 → 통계 계산 → InfluxDB 저장 + Grafana 시각화

### 하이브리드 I/O 모델
| I/O 모델 | 사용 시나리오 | 특징 |
|----------|-------------|------|
| **이벤트 기반 (epoll)** | 일반 (32~128 요청) | CPU 사용률 낮음, 대용량 데이터에 효과적 |
| **비지 폴링 (polling)** | 소량 요청 (<32) | 일관된 저지연, 소규모 데이터에 적합 |

- 두 모드를 결합하여 CPU 효율성과 지연 안정성 모두 확보

## 6. 사례 연구 (Case Study)

| 사례 | 설명 | ZERO+ 적용 |
|------|------|-----------|
| **Redis** | 200+ 메트릭/인스턴스. 기존: INFO 인터페이스 요청 → CPU 점유 | 할당 API로 메트릭 구조화 → 인스턴스당 READ 1회 |
| **Linux Kernel** | /proc 파일 읽기 → 유저/커널 공간 오버헤드 | per-CPU 카운터를 등록, 1~4 READ로 수집 (90%+ 호스트가 1~4 코어) |
| **eBPF** | TCP 연결 트래픽/재전송 모니터링. eBPF map syscall 빈번 | eBPF array map의 mmap 지원(Linux 5.5+)으로 에이전트에 직접 노출 |

## 7. 평가 결과

### 배포 환경 (4개 클러스터)

| 클러스터 | 노드 | 호스트 | RDMA NIC | 프로토콜 |
|---------|------|--------|----------|---------|
| Cluster 1 (테스트베드) | 8 PM | 1024 컨테이너 | Mellanox CX-4 | RoCEv1/2 |
| Cluster 2 (베어메탈) | 9 PM | 1024 컨테이너 | Mellanox CX-6 (100GbE×2) | RoCEv1/2 |
| Cluster 3 (경량 ECS) | 5 PM | 1024 컨테이너 | Alibaba eRDMA v1 | eRDMA v1 |
| Cluster 4 (강화 ECS) | 9 PM | 1024 컨테이너 | Alibaba eRDMA v2 | eRDMA v2 |

### 핵심 성능 결과

#### 오버헤드 비교 (ZERO+ vs 기존 시스템)

| 메트릭 | 기존(Netdata/NX) | ZERO+ RPC | **ZERO+** |
|--------|-----------------|-----------|-----------|
| **Redis 총 지연(ms)** | 0.7~19.3 | 0.08~0.18 | **0.05~0.14** |
| **Kernel 총 지연(ms)** | 0.5~1.6 | 0.14~0.36 | **0.07~0.23** |
| **eBPF 총 지연(ms)** | 0.8~12.5 | 0.10~1.02 | **0.08~0.87** |
| **에이전트 CPU (데이터 평면)** | 0.5~45% | 0.01~0.9% | **0%** |
| **에이전트 CPU (제어 평면)** | — | — | **0.04~1.5%** (일회성) |

#### 확장성 (대규모 모니터링)
- **1~10k 호스트**를 0.1~1s 샘플링 간격으로 **단일 스레드**로 수집 가능
- 대역폭 활용률: 여러 클러스터에서 **80~95%** 유지 (1k 호스트 모니터링 시)
- 흐름 제어 없이 256+ 호스트 → 처리량 대역폭의 10% 이하로 감소
- 흐름 제어 적용 시 1024 호스트 모니터링에서도 컨트롤러 CPU 약 **7~10%**

#### 대역폭 공유 FC vs Credit 기반 FC
- ZERO+ FC가 Credit-4KB/8KB/16KB 대비 **13~349% 높은 처리량** (클러스터별 상이)
- 고정 크레딧은 모니터링 규모/네트워크 환경의 변화에 적응 불가

#### 우선순위 큐
- 1k 호스트에서 고우선순위가 저우선순위보다 평균 **400μs 먼저** 완료
- 고우선순위 호스트의 꼬리 지연이 더 안정적

## 8. 운영 경험 및 교훈

| 주제 | 내용 |
|------|------|
| **확장성 & 가용성** | QP 공유(호스트당 1 QP) + QP 그룹핑(64 호스트 단위 전환). 에이전트당 3개 이상 컨트롤러 연결, 합의 기반 장애 감지 |

## 9. 관련 연구와의 차별점

| 분야 | 기존 연구 | ZERO+ 차별점 |
|------|---------|------------|

| 모니터링 시스템 | Ganglia, Netdata, Prometheus, Stackdriver | 모니터를 인프라에서 **디커플링**, 오버헤드 완전 제거 |
| RDMA 혼잡 제어 | DCQCN, HPCC, Swift, ExpressPass, Homa, RCC | One-sided RDMA의 CPU 바이패스 특성에 맞는 **수신자 주도 전략**, 애플리케이션 레이어에서 동작하여 기존 하드웨어에 즉시 적용 가능 |

## 10. 핵심 기여 요약

1. **최초의 제로 오버헤드 모니터링 시스템**: One-sided RDMA READ로 모니터링 대상 호스트의 CPU 점유율 0% 달성
2. **제어/데이터 평면 분리**: 일회성 제어 평면 오버헤드 + 완전 제로 오버헤드 데이터 평면
3. **대역폭 공유 기반 흐름 제어**: 클라우드 네이티브의 동적/블랙박스 네트워크 환경에 적응, 80~95% 대역폭 활용률
4. **우선순위 큐 QoS**: 지연 민감 서비스의 메트릭 수집 우선 완료 보장
5. **알리바바 클라우드 4개 이종 클러스터에 실제 배포 및 검증**
