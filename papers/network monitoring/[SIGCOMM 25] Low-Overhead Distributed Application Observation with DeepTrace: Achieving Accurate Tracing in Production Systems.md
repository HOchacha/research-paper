## 연구 배경 및 문제

### 마이크로서비스와 분산 트레이싱의 필요성
- 온라인 서비스가 모놀리식 → 분산 마이크로서비스 아키텍처로 전환
- 단일 사용자 요청이 수백 개의 컴포넌트 간 상호작용을 포함
- 장애 진단 및 성능 디버깅을 위해 **분산 트레이싱**(end-to-end 요청 흐름 추적)이 필수

### 기존 비침투적(Non-intrusive) 트레이싱의 한계
비침투적 트레이싱은 코드 수정 없이 동작하여 운영 효율성이 높지만, 두 가지 근본 문제 존재:

1. **FIFO 기반 상관관계 프레임워크** (예: vPath, DeepFlow)
   - 단일 스레드 순차 실행 모델을 가정
   - 동시성 처리 환경(코루틴, 비동기 I/O, 스레드 풀)에서 요청 매핑 모호성 발생
   - 고동시성 시 정확도 ~40%로 급락

2. **지연 기반 상관관계 프레임워크** (예: WAP5, TraceWeaver)
   - 부모-자식 요청 쌍의 지연 분포를 통계적으로 추정
   - 중앙 집중식 처리로 확장성 병목
   - 고동시성 시 후보 부모 요청이 증가하여 오매칭 확률 기하급수적 증가
   - TraceWeaver: 1500 RPS에서 63% 정확도

### 동시성 처리 메커니즘의 도전
- **Non-blocking I/O**: 스레드가 응답을 기다리지 않고 새 요청 처리 (예: Node.js)
- **코루틴**: 단일 스레드 내 여러 사용자 공간 코루틴 멀티플렉싱 (예: Go, Python)
- **Cross-thread**: 작업 큐/채널/세마포어를 통한 스레드 간 핸드오프 (예: Thrift, gRPC)

## 핵심 아이디어: 트랜잭션 기반 상관관계

### 트랜잭션(Transaction)의 정의
- **트랜잭션 카테고리**: 공유된 문맥 속성을 통해 의미적 일관성을 보이는 인바운드/아웃바운드 스팬(요청)의 논리적 관련 집합
- 결정 기준:
  - **중첩 API 엔드포인트 관계**: Login → VerifyID 같은 API 호출 의존성
  - **영속적 트랜잭션 필드**: 사용자 ID, 주문 정보, 액세스 토큰 등 요청 간 전파되는 필드

### 핵심 통찰
> 같은 트랜잭션에 속하는 스팬은 트랜잭션을 넘는 스팬보다 **더 강한 상관 가능성**을 보임

## DeepTrace 프레임워크 설계

### 설계 목표
| 목표 | 설명 |
|------|------|
| **Goal 1** | 다중 프로토콜 호환성 — 20개 이상의 애플리케이션 프로토콜 동시 지원 |
| **Goal 2** | 정밀한 end-to-end 트레이싱 — 높은 정확도로 부모-자식 관계 추론 |
| **Goal 3** | 고효율 운영 — 빠른 스팬 구축, 트레이스 재구성, 쿼리 응답 |
| **Goal 4** | 무시 가능한 성능 오버헤드 — 데이터 전송/저장 오버헤드 최소화 |

### 아키텍처 개요
- **에이전트(Agent)**: 각 호스트에 배포, 스팬 구축 및 부모-자식 관계 추론 수행
- **서버(Server)**: 마이크로서비스 클러스터 전체에서 운영, 사용자 쿼리에 따른 트레이스 조립

### Design 1: 프로토콜 인식 스팬 구축 (§3.1)

**eBPF 기반 비침투적 데이터 수집**
- 네트워크 관련 10개 시스템 콜(read, write, recvfrom, sendto 등)에 eBPF 프로그램 부착
- 수집 데이터: PID, 시스템 콜 타입, 소켓 5-tuple, 타임스탬프, 메시지 내용

**프로토콜 템플릿 기반 요청 파싱**
- RFC 문서 및 오픈소스 구현(Zeek, Suricata)에서 파생된 프로토콜 파싱 템플릿
- **20개 이상의 주류 애플리케이션 레이어 프로토콜** 지원
- 파싱 3단계:
  1. **전처리**: 타임스탬프 정렬 → PID/5-tuple/시스템콜 타입별 그룹화
  2. **요청 파싱**: 시그니처 기반 프로토콜 식별 + 점프 파싱(요청 길이 필드 활용)
  3. **요청-응답 연관**: 직렬/파이프라인/멀티플렉싱 모드별 매칭

**서드파티 통합**
- Kubernetes/클라우드 리소스 태그 주입 (노드, 서비스, Pod, VPC 등)
- Prometheus 메트릭 연동

### Design 2: 트랜잭션 분석 기반 부모-자식 확률 평가 (§3.2)

**중첩 API 엔드포인트 관계 추론**
- API 트래픽 강도 벡터 $X$ 구성 (각 quantity interval에서의 도착률)
- **피어슨 상관계수**로 API 쌍의 중첩 관계 확률 계산:

$$P_H(\cdot) = \frac{\sum_{i=1}^{n}(X_i - \bar{X})(Y_i - \bar{Y})}{\sqrt{\sum_{i=1}^{n}(X_i - \bar{X})^2} \sqrt{\sum_{i=1}^{n}(Y_i - \bar{Y})^2}}$$

**영속적 트랜잭션 필드 추론**
- **TF-IDF** 알고리즘으로 요청 내용의 필드 가중치 계산
- 크기-$s$ 슬라이딩 윈도우(기본 4바이트)로 내용 분할
- 스키마 필드(항상 등장)는 낮은 IDF → 자동 필터링
- **코사인 유사도**로 두 스팬의 트랜잭션 필드 공유 확률 $P_D(\cdot)$ 계산

**통합 확률 함수**
$$P(s, d) = w_1 \times P_H(s,d) + w_2 \times P_D(s,d) + \sum_{i=1} w_{i+2} \times P_M^i(s,d)$$

- $P_H$: API 엔드포인트 관계 확률
- $P_D$: 트랜잭션 필드 확률
- $P_M^i$: 다차원 메트릭(지연, 요청 크기, 지속시간) 분포 확률

### Design 3: 반복적 스팬 연관 + 자동 가중치 조정 (§3.2)

**Shannon 엔트로피 기반 가중치 조정**
- 각 확률 함수가 산출하는 확률값의 엔트로피 계산
- 엔트로피가 높아야 올바른/오류 매핑을 잘 구별 → 높은 가중치 부여
- 무효한 함수(예: 트랜잭션 필드 부재 시 $P_D$)는 자동으로 가중치 감소

**휴리스틱 선형 복잡도 탐색**
- 각 아웃고잉 스팬에 대해 최적 부모 인커밍 스팬 선택(greedy)
- 매 $k$개 매핑 선택 후 메트릭 분포 업데이트 (반복적 정제)
- NP-hard 다차원 할당 문제를 선형 시간에 근사 해결

### Design 4: 쿼리 기반 트레이스 조립 (§3.3)

**이중 인덱싱 전략**
- **역인덱스**: 태그 타입별 → 해당 태그를 포함하는 모든 스팬 참조
- **히스토그램**: 메트릭별 → 유사 메트릭값 스팬 그룹화
- 저장 정보 최소화: 부모-자식 관계 + 타임스탬프만 저장

**온디맨드 스팬 수집**
- 서버가 에이전트에 쿼리 → 관련 스팬 매핑 수집 → 반복적으로 부모/자식 확장
- 전체 스팬 중앙 수집 불필요 → 전송 오버헤드 94% 감소

## 프로덕션 배포 사례

### Use Case 1: Tail Latency 진단 (금융기관)
- **환경**: OpenStack(가상화) + Kubernetes(마이크로서비스), HTTP/gRPC/DB 프로토콜
- **문제**: 클라이언트 요청 타임아웃, Prometheus로 개별 요청 수준 분석 불가 → 수일간 미해결
- **DeepTrace 적용**: 코드 수정 없이 에이전트 배포 → 타임아웃 트레이스 쿼리 → **15분** 내 근본 원인 컴포넌트 식별 (연결 리소스 부족)

### Use Case 2: Application DDoS 분석 (스마트 기기 제조사)
- **환경**: 멀티 AZ 퍼블릭 클라우드, APISIX 게이트웨이, K8s 클러스터
- **문제**: 저빈도(150 RPS) 깊은 중첩 쿼리 파라미터로 Redis 리소스 고갈 → 기존 모니터링 감지 불가
- **DeepTrace 적용**: Redis 응답 시간 15ms → 210ms 변화 감지 → 분 단위 내 공격 대상 컴포넌트 특정 → 방화벽 규칙 적용

## 테스트베드 평가 결과

### 실험 환경
- 3노드 Kubernetes 클러스터 (Intel Xeon E5-2620 v3, 128GB RAM, Ubuntu 22.04)
- 워크로드: DeathStarBench의 Social Networking(36 컴포넌트) + Movie Reviewing(38 컴포넌트)
- Ground truth: Jaeger(침투적 트레이싱)로 확인

### 주요 결과

| 평가 항목 | DeepTrace 결과 | 비교 |
|----------|--------------|-----|
| **부하별 정확도** | 모든 RPS에서 **95% 이상** | DeepFlow/vPath: 1500 RPS에서 ~40%, TraceWeaver: 63% |
| **Tail 요청 정확도** | tail 10%ile에서 **92%** | 모든 baseline이 tail 요청에서 오류 집중 |
| **지연 변동 내성** | 50% 지연 주입 시에도 **95%** | TraceWeaver: 76% |
| **동적 실행 경로** | 50% 캐시 비율에서 **97%** | TraceWeaver: 70% |
| **트레이스 재구성 속도** | 10K 요청 ~**40초** | TraceWeaver: ~440초 (**10배 이상 빠름**) |
| **전송 오버헤드** | 30~60 MB | 샘플링 없는 프레임워크: ~1000 MB (**94% 감소**) |
| **성능 영향** | 처리량 **4.6% 감소** (1,667 RPS) | Jaeger 100%: 24% 감소, DeepFlow: 유사 수준 |

### Ablation Study (1500 RPS)
| 제거 모듈 | 정확도 감소 |
|----------|----------|
| 자동 가중치 조정 | -2% |
| 중첩 API 정보 | -1% |
| 트랜잭션 필드 | **-9%** (가장 큰 영향) |
| 메트릭 (지속시간, 요청 크기) | -2% |

## 향후 방향 및 Lessons Learned

1. **글로벌 추론**: 현재 컴포넌트별 독립 추론 → 50개 컴포넌트 트레이스 시 end-to-end 정확도 61%까지 하락 ($0.99^{50}$) → 글로벌 추론으로 개선 계획
2. **스레드 정보 활용**: dispatcher-worker 모델 등 스레딩 패턴 활용 가능
3. **부하 변동 활용**: 저부하 시기의 정확한 매핑으로 메트릭 분포 고도화

## 비교: 기존 프레임워크와의 차이

| 특성 | FIFO 기반 (DeepFlow, vPath) | 지연 기반 (TraceWeaver, WAP5) | **DeepTrace (트랜잭션 기반)** |
|------|--------------------------|---------------------------|--------------------------|
| 코드 수정 | 불필요 | 불필요 | **불필요** |
| 동시성 처리 | ❌ 순차 모델 가정으로 실패 | △ 이론적 대응 가능 | **✅ 트랜잭션 의미론 활용** |
| 정확도 (고동시성) | ~40% | ~63% | **95%+** |
| 확장성 | 좋음 | 중앙 집중 병목 | **에이전트 로컬 처리** |
| 성능 오버헤드 | ~24% | N/A | **~5%** |
| 프로토콜 지원 | 제한적 (HTTP, gRPC) | 제한적 | **20+ 프로토콜** |

## 핵심 기여 요약

1. 분산 트레이싱의 스팬 상관관계 설계에 대한 **트랜잭션 관점의 새로운 시각** 제시
2. eBPF 기반 **20개 이상 프로토콜 지원**의 비침투적 스팬 구축
3. **TF-IDF + 피어슨 상관 + 엔트로피 기반 자동 가중치** 조정의 트랜잭션 기반 스팬 연관 메커니즘
4. **쿼리 기반 트레이스 조립**으로 전송 오버헤드 94% 감소
5. 수십 개 기업 프로덕션 환경에서 실제 배포 및 검증
