## 연구 배경 및 문제

### Alibaba Cloud 네트워크 인프라 규모
- 2025년 1월 기준, 29개 지리적 리전에 걸쳐 87개 데이터센터 운영
- 프라이빗 WAN이 백본으로 모든 데이터센터 네트워크를 상호 연결
- 주간 평균: 2,377건의 이상(anomaly)은 자동 치유(self-healing), 202건은 운영자가 직접 처리(인시던트)

### 인시던트 관리(IM) 워크플로
```
네트워크 모니터링 → 이상 탐지 → 자동 치유(~90%) → 수동 조사(~10%) → 인시던트 완화
```
- 자동 치유: 사전 정의된 규칙과 운영자 경험 기반 자동화 솔루션으로 처리
- 수동 조사: 자동 치유 실패 시 온콜(on-call) 운영자가 개입하여 근본 원인 파악 및 장애 장치(error device) 식별

### 장애 위치 파악(Failure Localization)이 어려운 이유

| 도전과제 | 설명 |
|---------|------|
| **과다한 알림** | 단일 인시던트의 모니터 알림 총량이 평균 26.4MB, 최대 1GB(8k 로그 항목) 이상. 수 대~수백 대의 장치에 걸쳐 수백 건의 알림 발생 |
| **복잡한 근본 원인 분석** | 시간 압박 하에 모든 정보를 분석하는 것이 비현실적. 근본 원인 파악에 10분 이상 소요(SLA 초과), 최장 30분 이상 |
| **높은 인시던트 부하** | 주당 약 202건의 인시던트를 처리해야 하며, 효율성과 정확성이 운영자의 전문성·경험에 크게 의존 |

### 동기 부여 예시 (Figure 3)
- 데이터센터에서 대규모 패킷 손실 알림 감지 → 4분 내에 다수 인시던트 보고 (B2-A1 플래핑, B3-A1 플래핑, B2 핑 불가, ICMP 알림)
- 운영자가 서비스 업데이트 의심 → 배제 → 트래픽 감소에 초점 → 22분 후에야 B2를 근본 원인으로 확인하고 격리 완료
- 대량의 알림 속에서 근본 원인을 체계적으로 찾기가 매우 어려움

## LLM 활용의 기회와 기존 연구의 한계

### LLM이 적합한 이유
- 모니터링 도구의 출력이 주로 텍스트 기반이며, 인간 운영자가 해석하도록 설계됨
- LLM은 대량 텍스트의 효율적 읽기·추론에 탁월하고, 피로감 없이 지속 운영 가능
- 규칙 기반·통계적 방법 대비 논리적 추론과 미경험 사례에 대한 일반화 능력이 우수

### 기존 LLM 기반 연구의 한계

| 기존 연구 | 한계 |
|----------|------|
| **MonitorAssistant** [81] | 가이드 지향 이상 보고서만 생성, 근본 원인 장치 미식별 |
| **Oasis** [33] | 장애 영향 범위 평가 및 요약만 제공 |
| **RCACopilot** [16] | 근본 원인 *카테고리*만 출력, 구체적 장애 장치 미식별 |
| **NetAssistant** [65] | 진단 의도 인식 후 사전 정의 워크플로 실행, 조사는 운영자에게 위임 |
| **Ahmed et al.** [5] | 인시던트 제목·요약만으로 근본 원인 분석 → 상세 로그 없이는 정확한 판단 불가 |

→ 기존 연구는 **coarse-grained 분석**이거나 **부분 정보만 활용**하여, 구체적 장애 장치를 직접 식별하지 못함

## BiAn 시스템 개요

### 목표
- 대규모 네트워크 인프라(WAN + 데이터센터)에서 **빠르고 정확한 근본 원인 분석 및 장애 위치 파악**
- 기존 모니터링 도구를 대체하지 않고, 운영자의 인시던트 조사를 **보조**하는 어시스턴트 역할

### 입출력
- **입력**: 후보 장치(upstream 자동화 시스템이 선정한 가장 의심스러운 ~6대)에 대한 모니터 알림
- **출력**: 후보 장치의 **장애 확률 순위 + 설명**으로 구성된 랭킹 리스트
- 운영자가 BiAn의 출력을 검토하고 최종 결정

### 시스템 아키텍처 (Figure 4)


## 핵심 설계

### 4.1 계층적 추론 (Hierarchical Reasoning)

대량의 모니터링 데이터를 한 번에 처리하는 것이 LLM에게도 한계가 있으므로, 3단계로 분해:

#### Step 1: Monitor Alert Summary
- 11개 모니터링 도구의 알림을 각각 전담 LLM 에이전트가 처리
- 각 장치에 대해 11개의 **축약된 요약**을 생성
- 프롬프트 구조: 역할 정의 / 입력 필드 설명 / 요약 가이드라인 / 알림 예시 + 기대 요약 / 응답 형식

#### Step 2: Single-device Anomaly Analysis
- 운영자 10년 경험 기반으로 정의된 **7가지 이상 시나리오**:

| 이상 시나리오 | 사용 데이터 소스 (Figure 5) |
|-------------|-------------------------|
| **Device Down** | Dashboard Alarms, Link States, Device Ping Log |
| **Congestion** | Link Traffic Usage |
| **Traffic Drop** | Traffic Drop Device |
| **Flapping** | Dashboard Alarms, Port Down Events |
| **Network Changes** | Command History, Device Change History |
| **Syslog Surge** | High-risk Syslog, Syslog Volume |
| **Alarm Count** | Dashboard Alarms |

- 7개 시나리오로 알림을 통해 관찰 가능한 인시던트의 **100%를 커버**
- 각 시나리오를 전담하는 7개 LLM 에이전트가 SOP(Standard Operating Procedure)에 따라 분석

#### Step 3: Joint Scoring
- 모든 의심 장치의 이상 분석 보고서를 통합하여 **장애 점수(failure score)** 산출 (합계 = 1)
- 핵심 통찰: **장애 장치는 다른 장치 대비 더 심각하고 더 많은 이상 행동**을 보임
- 최고 점수 장치 = 장애 장치로 마킹

---

### 4.2 3-파이프라인 통합 (Three-pipeline Integration)

Pipeline 1만으로는 복잡한 공간적·시간적 관계를 충분히 파악할 수 없는 경우를 위해 2단계 추론:

#### Pipeline 2: Network Topology
- 모든 의심 장치를 포함하는 **서브 토폴로지** 추출
- 알고리즘:
  1. 모든 장치 쌍의 최단 경로 탐색
  2. 의심 장치 C가 A-B 최단 경로에 존재하면 A-B 경로 제거 (A-C, B-C로 대체)
  3. 남은 최단 경로로 토폴로지 구성
  4. 동일 장치 그룹의 비의심 장치를 단일 노드로 집약
- 부모 장치에 이상이 있으면 자식 장치의 이상도 전파된 것으로 추론 가능

#### Pipeline 3: Event Timeline
- 모든 장치의 이벤트를 시작 시간 순으로 정렬한 타임라인
- **먼저 보고된 이상을 가진 장치일수록 의심도 상승**

#### Integrated Root Causing
- Stage 1 완료 후 **Top-p 필터**로 낮은 의심도 장치 제거 (softmax 적용 후 누적 확률이 p에 도달하는 장치만 유지)
- 3개 파이프라인(장치 이상 보고서 + 토폴로지 + 타임라인)을 통합한 최종 추론 에이전트 실행
- 프롬프트 구조 (Figure 6): 역할 정의 / 이상 분석 보고서 / 네트워크 토폴로지 / 글로벌 타임라인 / 조사 원칙(SOP)

#### Rank of Ranks (결과 안정성 확보)
- LLM의 무작위성 완화를 위해 통합 추론을 **N=3회** 반복 실행
- 각 실행에서 장치별 순위 산출 → N회의 **평균 순위**로 최종 장애 장치 결정
- 단순 점수 평균 대비 이상치(outlier)에 강건함
  - 예: 세 번 시행에서 두 장치 점수가 {0.6, 0.3}, {0.5, 0.4}, {0.2, 0.7}일 때, 점수 평균은 역전되지만 Rank of Ranks는 올바른 판단 유지

### 4.3 지속적 프롬프트 갱신 (Continuous Prompt Updating)

네트워크 진화에 따라 LLM 에이전트의 추론 프롬프트를 지속적으로 "훈련"하는 메커니즘 (Figure 7):

| 단계 | 설명 |
|------|------|
| **Exploration** | 현재 프롬프트로 인시던트당 5회 추론 시도 → 100% 정확도 사례 제외 → 나머지를 높은 Temperature T로 재시도하여 정답/오답 혼합 결과 생성 (Zero-shot CoT 사용) |
| **Reflection** | Agent R이 정답/오답 시행을 대조 분석하여 핵심 영향 요인과 실용적 지식(actionable knowledge) 추출 |
| **Knowledge Consolidation** | Agent C가 모든 인시던트에서 추출된 지식을 요약, 빈도 카운트 후 **상위 6개** 가장 빈번한 지식 유지 |
| **Prompt Augmentation** | 버퍼 배열에 새 지식 추가·병합 → LLM이 기존 프롬프트 + 지식 버퍼로 "정제된 프롬프트" 생성 |

- 수개월마다 주요 하드웨어·소프트웨어 업그레이드에 맞춰 프롬프트 갱신 수행
- 한계: 프롬프트 길이 증가 → 일정 수준 이상에서 성능 저하 발생 (프롬프트 "훈련"의 근본적 한계)

### 4.4 시스템 최적화

| 최적화 | 설명 |
|--------|------|
| **Fine-tuning** | Pipeline 1의 단순 태스크(알림 요약, 단일 장치 이상 분석)에 **Qwen2.5-7B** 파인튜닝. GPT-4로 합성 데이터 생성 후 규칙 기반 검증. 정확도 향상 + 추론 지연 감소 |

| **Parallel Execution** | 동일 단계 내 에이전트를 동시 실행 (알림 요약 에이전트, 이상 분석 에이전트, Rank of Ranks 복수 실행 등) |

## 실제 배포 결과 (10개월 운영)

### A/B 테스트 (Figure 8a)
- **TTR(Time-to-Root-Causing)**: 조사 시작 ~ 근본 원인 및 장애 장치 식별까지의 시간

| 리스크 수준 | Human (수동) | RCACopilot 보조 | BiAn 보조 |
|-----------|-------------|----------------|---------|
| Low | 기준 | 약간 단축 | 단축 |
| Medium | 기준 | 일부 단축 | 단축 |
| High | 기준 | 소폭 단축 | **대폭 단축** |

- BiAn은 **평균 TTR 20.5% 감소**, 고위험 인시던트에서 **55.2% 감소**
- RCACopilot은 근본 원인 카테고리만 제공하므로 운영자가 여전히 모니터링 데이터를 재검토해야 함

### 설명 가능성 (Figure 8b)
- 운영자가 BiAn의 설명을 0(도움 안 됨) / 1(다소 도움) / 2(매우 도움)으로 평가
- 대부분 1 또는 2로 평가, 0은 극소수
- Top-1 장치가 틀린 경우에도 설명이 유용한 맥락이나 중간 추론을 제공

### 사례 연구

#### Incident 1 (동기 부여 예시 재활용)
- BiAn: B2를 가장 의심스러운 장치(4개 이상 감지), B3가 2순위(3개 이상) → **28초 만에 장애 장치 식별**
- 운영자: 동일 데이터 접근에도 22분 소요

#### Incident 2 (토폴로지 파이프라인의 효과)
- Stage 1에서 A1, A2가 동일 점수 (둘 다 traffic drop + flapping)
- Stage 2에서 토폴로지 파이프라인이 **A2가 B1의 부모 노드**임을 반영 → A2를 높은 점수로 식별
- 근본 원인: A2의 라인카드 고장. TTR = **3분**

#### Incident 3 (타임라인 파이프라인의 효과)
- A1(disconnection + network changes), A2·B1(traffic drop) → 초기 유사 점수
- 타임라인에서 **A1의 알림이 가장 먼저 발생** → A1을 장애 장치로 판정. TTR = **1.5분**

## 성능 평가 (17개월간 실제 인시던트 357건)

### 모델 구성
- Pipeline 1 Step 1~2: **Qwen2.5-7B-Instruct** (파인튜닝)
- 기타 모든 태스크: **Qwen2.5-72B-Instruct**

### 정확도 (§6.1)

| 메트릭 | BiAn | Hot Device (베이스라인) |
|--------|------|----------------------|
| **Top-1 정확도** | **95.5%** | 86.3% (tied 시 70.5%) |
| **Top-2 정확도** | 98.6% | 88.9% |
| **Top-3 정확도** | 99.3% | 93.0% |

- Hot Device: 관련 알림이 가장 많은 장치를 선택하는 "민주적" 방식 (007 [7] 적용)
- BiAn은 리스크 수준, 난이도, 장애 유형 모든 카테고리에서 Hot Device를 상회 (Figure 9)
- 라인카드 장애에서 최고 성능 (장애가 호스팅 장치에만 영향 → 이웃 장치 이상이 적어 구별 용이)

### 지연 시간 (§6.2)
- 전체 추론 과정이 **30초 이내** 완료 (Figure 10)
- 인시던트 발생 → 자동 시스템이 운영자에게 알리고 정보 로딩(~1분) → 운영자 접속 시 BiAn 결과 이미 준비
- 파인튜닝된 소형 모델로 Stage 1의 알림 요약 및 이상 분석 지연 대폭 감소 (Figure 11)

### Ablation Study (§6.3, Figure 12)

| 구성 | Top-1 정확도 |
|------|------------|
| Stage 1 only (Pipeline 1) | 87.2% |
| + Top-p 필터 | 93.1% |
| + Top-p + Pipe.2 (Topology) | 더 높음 |
| + Top-p + Pipe.3 (Timeline) | 더 높음 |
| **+ Top-p + Pipe.2 + Pipe.3 (전체)** | **95.5%** |
| Hot Device | 86.3% |

- Top-p 필터 없이 3개 파이프라인 실행 시 오히려 정확도 감소 → 불필요한 장치 데이터가 모델 주의력(attention) 희석

### Early Stop 성능 트레이드오프 (§6.3, Figure 13)
- 엔트로피 임계값 0.75에서 전체 사례의 약 1/3이 Stage 1에서 조기 종료
- 처리 시간 70%로 감소, 정확도 손실 0.5%에 불과

### 결과 안정성 (§6.4, Figure 14)
- Rank of Ranks (N=3): 평균 정확도는 단일 실행과 유사하나, **분산이 현저히 감소**
- N=3 → N=5로 증가해도 유의미한 차이 없음

### 파인튜닝 효과 (§6.5, Figure 15)
- 알림 요약: 파인튜닝 모델 **98.7%** vs 기본 모델 ~90%
- 이상 분석: 파인튜닝 모델 **98.6%** vs 기본 모델 ~92%

### 프롬프트 갱신 효과 (§6.5, Figure 16)
- 3회 반복 후: 훈련 정확도 54.0% → 62.0%, 테스트 정확도 36.7% → 50.0% (이미 100% 정확 사례 제외 후)
- 과도한 반복 시 프롬프트 길이 증가로 성능 저하

## 교훈 및 논의

### API 지연 변동
- Qwen 온라인 서비스의 추론 지연 평균 ~5초이나 개별 호출은 최대 16초까지 변동
- 지연 민감 태스크에 영향 가능 → 로컬 배포(dedicated resource) 방식 고려

### 설명 가능성과 신뢰
- 설명 포함이 정확도에 영향 없으나, **점수 엔트로피를 감소**시킴 (예: {0.5, 0.3, 0.2} → {0.7, 0.2, 0.1})
- 잘못된 순위에 유용한 설명이, 또는 올바른 순위에 잘못된 설명이 포함될 수 있음

### 프롬프트 엔지니어링
- 프롬프트 길이 관리가 핵심 → 길어질수록 성능 저하
- LLM이 분석 단계를 건너뛰거나 과도하게 장황한 출력을 생성하는 경우 발생
- 제어(control)와 유연성(flexibility) 사이의 트레이드오프가 비자명
- 사용 기법: few-shot prompting, chain-of-thought, self-consistency

### Multi-agent 시스템
- 초기에 LLM 에이전트가 자유롭게 분석 태스크를 계획하도록 시도 → 반복 루프, 미종료, 과도한 컨텍스트 축적 문제 발생
- **구조화된 계층적 프레임워크**(사전 정의 역할 + SOP 기반)로 전환하여 제어 가능성 확보
- 향후: 재시도, 단계 건너뛰기, 동적 에이전트 호출 등 더 적응적 행동 필요

### 한계
1. **다중 장치 장애 미지원**: 인시던트당 단일 장애 장치를 가정. 드물지만 2대가 동시 장애인 경우 존재 (다만 상위 2개 점수가 근접하여 운영자가 식별 가능)
2. **업스트림 모니터 의존**: 모니터가 유효하지 않거나 모호한 데이터를 생성하면 정확한 결과 제공 어려움
3. **링크 기반 인시던트 미지원**: 현재 장치 기반 인시던트에 집중. 링크 기반 인시던트는 다른 조사 프로세스와 SOP 필요 → 확장 예정
